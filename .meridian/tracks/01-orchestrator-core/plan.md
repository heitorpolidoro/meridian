# Implementation Plan: Meridian Project Orchestrator (Core)

**Status:** `Final RFC`
**Reference Specification:** `.meridian/tracks/01-orchestrator-core/spec.md`

## 1. Proposed Architecture

The Meridian Project Orchestrator Core is built on a hierarchical "Agent Bootstrapping" model, designed for context isolation and strict adherence to the **Software Development Standard (SDS)**.

### 1.1 System Communication Flow
The following diagram illustrates the interaction between the user interface, the orchestration engine, and the underlying AI agents.

```text
                                 +-----------------------+
                                 |     User (React UI)   |
                                 |   (Vite + Vanilla CSS)|
                                 +-----------^-----------+
                                             |
                                             | Socket.io (Real-time Streaming)
                                             |
                                 +-----------v-----------+
                                 |   Meridian Backend    |
                                 |  (Express + Node.js)  |
                                 +-----^-----------+-----+
                                       |           |
                 (1) Bootstrapping     |           | (2) LLM Adapter (ACP)
              Reads Roles/Standards    |           |     Spawn & Stream
                                       |           |
      +-----------------------+        |     +-----v-----------------+
      |   Local Filesystem    +--------+     |   LLM Engine (Agent)  |
      | (.meridian/ & agents_)| <------------+ (Execution Context)   |
      +-----------------------+  (3) I/O     +-----------------------+
```

### 1.2 Tech Stack Consolidation
To ensure high performance and local-first reliability, the following stack is formalized:

| Layer | Technology | Rationale |
| :--- | :--- | :--- |
| **Frontend** | React + Vite + Vanilla CSS | Clean, high-performance UI without the overhead of external frameworks (KISS). |
| **Backend** | Node.js + Express | Lightweight, event-driven architecture suitable for handling multiple subprocesses. |
| **Real-time IPC**| Socket.io | Enables bi-directional, low-latency streaming of agent logs and status updates to the UI. |
| **LLM Layer** | LLM Adapter (ACP) | Standardized communication with the LLM Engine via standard I/O streams. |
| **Storage** | Local Filesystem | Uses Markdown (`.md`) for documentation and JSON for metadata to ensure transparency and portability. |

### 1.3 IPC Protocol: Real-time Streaming
Socket.io is the backbone of the human-orchestrator interaction. It facilitates:
- **Live Output Streaming:** Every token generated by the LLM Engine is pushed to the UI immediately.
- **Bi-directional Control:** Allows the user to interrupt, pause, or send manual directives to the agent mid-execution.
- **Status Heartbeats:** Real-time telemetry (CPU/Memory usage of agents) is broadcasted to the dashboard.

### 1.4 Bootstrapping Logic (Priming)
The "Knowledge Isolation" core principle is enforced through a strict bootstrapping sequence:
1. **Standard Loading:** The Backend reads `agents_core/core/global.md` and relevant technical cores (e.g., `backend.md`).
2. **Role Activation:** The specific agent role (e.g., `agents_core/roles/architect.md`) is loaded based on the active track's requirements.
3. **Session Priming:** These standards and instructions are concatenated into a high-priority "System Instruction" block.
4. **Context Injection:** This block is passed to the LLM Engine via the LLM Adapter during the `spawn` event, ensuring the agent is "primed" with Meridian's identity and SDS guardrails before any user input is processed.

### 1.5 LLM Adapter Pattern (Abstraction)
To ensure the orchestrator is decoupled from specific AI providers, the backend interacts with the LLM via a standardized **LLM Adapter/Connector** pattern.
- **Interface Stability:** The Backend only speaks to the Adapter interface, which handles the specifics of the underlying engine.
- **Current Implementation:** Uses the ACP (Agent Control Protocol) over standard I/O to communicate with the local LLM Engine.
- **Future-Proofing:** This allows swapping the local engine for cloud-based APIs (e.g., Gemini API, OpenAI) or other local runners by simply implementing a new Adapter, without changing the core orchestration logic.

### 1.6 Service Abstraction & Mocking
To ensure 100% test coverage (SDS 4.1), all external interactions (Filesystem, IDE APIs, Shell) are abstracted behind service interfaces. This allows for total mocking during unit testing and ensures core logic remains decoupled from the environment.

### 1.7 Track Navigator Technical Design
The Track Navigator provides a structured interface for browsing and selecting Meridian Tracks and their documentation.

- **Backend Event: `get-tracks`**
  - **Logic:** Recursively scans the `.meridian/tracks/` directory.
  - **Output:** Returns a JSON tree structure of folders and `.md` files (e.g., `spec.md`, `plan.md`, `tasks.md`).
  - **Purpose:** Populates the sidebar navigation in the React UI.

- **Frontend State Management**
  - **`selectedTrack`**: React state holding the ID or folder name of the currently active track.
  - **`selectedFile`**: React state holding the relative path of the documentation file currently being viewed.
  - **Interactivity:** Updates to these states trigger re-renders of the navigation tree and the `MarkdownViewer`.

- **Markdown Integration**
  - **Request Flow:** When `selectedFile` changes, the frontend sends the file path to the backend via a socket event (e.g., `read-file`).
  - **Response:** The backend fetches the raw Markdown content and returns it to the UI for rendering within the `MarkdownViewer` component.

- **Security: Path Sanitization**
  - **Scope Enforcement:** The backend implements strict path sanitization to ensure only files within the `.meridian/tracks/` directory can be accessed.
  - **Verification:** All file requests are resolved relative to the tracks root and verified to remain within that boundary (preventing directory traversal).

## 3. Security (Practical Security - KISS)

### 3.1 Secret Prevention
Meridian adopts a "Security by Absence" strategy for local credentials.
- **Environment Isolation:** All API keys and environment-specific settings are stored in `.env` files.
- **Source Control Protection:** A strict `.gitignore` policy prevents any `.env` or temporary credential files from being committed.
- **No Local Vaults:** By avoiding complex local encryption (like AES-256 or `node-keytar`), we reduce implementation complexity and the risk of "bricked" local states.

### 3.2 Context Isolation
The primary security layer is **Task-Level Context Injection**. 
- **Minimum Viable Context:** Agents are only provided with the files and metadata explicitly linked to their active track.
- **Isolation by Default:** The Orchestrator core ensures that an agent working on "Track A" cannot see the implementation details or secrets of "Track B" unless explicitly granted.

### 3.3 IPC & Input Security
- **Schema-Validated IPC:** All communication between the UI (Frontend) and the Orchestrator Server (Backend) must be validated against a strict JSON Schema/Zod definition.
- **Input Sanitization:** User directives and agent outputs are sanitized to prevent injection attacks (e.g., shell injection).

## 4. Health Telemetry Service (SRE)

A dedicated service to provide real-time observability into the Orchestrator's health and SDS compliance.

### 4.1 AI Reliability Metrics
- **Latency:** Tracking P50 and P95 response times for LLM interactions.
- **Consumption:** Token usage per agent/track.
- **Error Rates:** Tracking model-specific errors and retries.

### 4.2 Data Integrity & Sync
- **Watcher Service:** Monitors the consistency between the local filesystem (`.meridian/`) and the Orchestrator's in-memory state.
- **Conflict Resolution:** Alerts the user if manual file changes conflict with agent-driven updates.

### 4.3 SDS Visibility
- **Compliance Score:** Percentage of active tracks that have valid `spec.md`, `plan.md`, and `qa_signoff.md`.
- **Target Tracking:** Coverage reports from CI integrated into the UI.

## 6. Business Value & Lifecycle Management

### 6.1 ROI Analysis
Centralized orchestration reduces cognitive load and ensures SDS compliance by default, reducing the "Cost of Inaction" (technical debt, manual sync).

### 6.2 Pivot & Termination Triggers
- **Pivot to CLI:** If UI development slows down orchestration speed, the core logic remains decoupled and can be exposed via a terminal-first interface.
- **Emergency Stop:** If metadata corruption is detected by the Telemetry Service, the orchestrator will lock further writes and prompt for a manual restore from the last known good state.
